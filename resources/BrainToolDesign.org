#+title: Brain Tool Design
* Introduction
This file gives an overview of the major components of the BrainTool browser extension. From a users perspective BrainTool (or BT) has a number of components. The Topic Manager runs in a tab, panel window or the browser sidepanel. It shows a hierarchy of 'topics' and contained links to web pages, plus notes and to-do annotation. The Bookmarker is invoked when the extension icon is clicked. It allows the current tab, tabgroup or page to be saved into the Topic Manager. The Sidepanel is just a shell which runs the topic manager in an iframe in the browsers sidepanel. Behind the scenes the service worker code deals with browser interaction, forwards events to the topic manager, and performs actions on the topic managers behalf.
The Topic Manager code is served from the braintool.org site hosted on github. It saves user data in org-mode format to browser local storage with optional file backup.

* General
The BT codebase is plain vanilla javascript, html and CSS. The Topic Manager makes heavy use of JQuery and the jquery treetable widget. It also uses a forked version of orga.js to parse org-mode and a css library called wenk.css for mouse over messages. The extension code does not use jquery or any libraries other than the awesomeplete autocompletion code.

* Components
BrainTool is composed of extension code that runs inside the browser, app code that is served from the github repository and a set of markdown files and resources which compose the braintool.org web site.

** Extension
Extension code is all contained in the /extension directory. This directory is compressed and uploaded to the browser app store. The extension code is modularized with import/export annotations.
*** Service worker
The service worker code is mainly contained in background.js. This code registers for events and forwards them to the application. It also listens for messages from the app and sends back required information.
*** Bookmarker popup
The popup.* files and the topic*.js files handles the Bookmarker interaction.
*** Content Script
The content script code is injected into the tab running the Topic Manager (served from braintool.org). It relays messages between the extension and the application.
*** SidePanel
The sidepanel.* js html code provides a slim wrapper that runs the topic manager in an iframe and mediates communication.

** App
The application code is all contained in the /app directory. As of Phase 2 refactoring (completed), the code has been successfully modularized with explicit ES6 import/export modules. The main bt.js file has been broken down into focused modules:
- applicationUI.js: Generates all UI chrome (header, footer, settings, actions, help panels)
- messageManager.js: Handles intro slideshow and message banners with HTML generation
- rowManager.js: Handles all row operations and button row UI generation
- tableManager.js: Handles table generation and UI initialization

*** bt.js
This file now focuses on core application logic:
**** Launch activities and startup orchestration

**** Tab/TabGroup/Window state management (tabOpened, tabClosed, tabNavigated, etc)
**** Search and Filter operations
**** Keyboard event handling
**** Session management and stats tracking
*** BTNode, BTAppNode
The BT Topic Manager table shows a hierarchy of 'nodes' where each node corresponds to a heading in the underlying org-mode data structure and to a row in the treetable in the UI. Each node models either a bookmark or saved web page (aka 'items'), or a topic which contains a set of saved items. Node items may also have an associated open tab in the browser. Topics are shown in the browser as tab groups and so topic nodes may also have an associated open tab group. A global AllNodes array holds the nodes. It is created on startup from the org data loaded in from local storage (or backup file) and may be sparse (due to item deletion).

The BTNode.js defines the base class of BTNode which handles containment and basic node operations.

The BTAppNode.js file handles the BTAppNode subclass of BTNode which handles all the operations of the nodes in terms of their being org-mode items in a file and being saved and potentially open tabs and tab groups, as well as being rows in the topic table.

There is also a BTLink subclass which represents a url embedded in the org data but not structured as an org header. This is a side effect of trying to support the import of any org-structured file and subsequently showing a clickable link in the tree. IT does not arise in normal BT usage and can generally be ignored.
*** parser.js orga-bundlev2.js
The orgabundlev2.js file contains the org-mode parsing code. The parser.js file is a wrapper which uses the orga code to read org text and create BTNodes and to write org to save BTNode data.
*** jquery.treetable
This is a forked version of the jquery treetable widget which handles the basic operations of creating and displaying the topic tree.
*** fileManager.js, localFileManager.js, gDriveFileManager.js
These files handle reading and writing the BT saved data, in org format, to a local or GDrive file. As well as providing notifications when the files may be out of date.
*** configManager.js
Handles preference/settings and config operations
*** bookmarksManager.js
Handles the import and export of browser bookmarks and the interaction with the browser bookmarks bar.
*** messageManager.js
Handles the display of intro slides that are shown on first install and banner messages that are shown on startup and when a new version is served.
*** subscriptionManager.js
Handles the interaction with a backend Firestore server which integrates with Stripe to allow permanent license purchase and monthly or annual subscriptions.
*** index.html
Entry point for the application. Now minimal with most UI generated dynamically:
**** Static content: #loading, #backgroundLogo, #content table, #editOverlay with dialogs
**** Generated by modules: All headers, footers, panels, message containers, intro slideshow, button row
**** Initialization script: Imports and calls initialize functions from applicationUI, messageManager, rowManager
**** Window shim: Minimal - only 3 edit dialog handlers remain (closeDialog, cancelEdit, updateRow - will be removed in Phase 3)
**** Module loading: bt.js loaded last after UI components initialized
*** bt.css, jquery.treetable.css, wenk.css
Between them these files contain all the application css. wenk.css is a standalone library for toolips.

* Build Process
The extension is manually bundled into a tar file and submitted to the store. To be able to handle different versions of the extension codebase simultaneously across users the extension specifies a version and the /app code specific to that version is loaded into the Topic Manager tab when its launched. While /app and /extension have the latest code the /versions hierarchy has the supported production versions. For the purposes of new development and examining the codebase LLM's should ignore the /versions hierarchy in order to avoid expired conventions or old message APIs etc.

At runtime when the Bookmarker popup is initially clicked the index.html file is loaded and orchestrates the loading of the rest of the app via <script> statements

** Phase 6 - Finish extension/'s ES6 module migration
The Phase 1 migration covered app/ only. extension/ is still half-migrated: background.js, bookmarkHandler.js, config.js, configFallback.js and summarizer.js are ES modules, but popup.js, sidePanel.js, btContentScript.js, summarizerPopup.js, topicCard.js and topicSelector.js are still classic scripts that share one global scope via `<script>` tag load order per HTML page (popup.html loads awesomplete.js+topicSelector.js+topicCard.js+popup.js; summarizerPopup.html loads only topicCard.js+summarizerPopup.js; sidePanel.html loads btContentScript.js+sidePanel.js).

*** Motivation
Setting up ESLint (2026-07-17, see de6c7a4) required splitting its config into a separate globals block per HTML page's actual script list, rather than one shared pool for all of extension/ - treating them as one pool would have masked exactly the kind of bug it's meant to catch. It did catch two: `summarizerPopup.js` references `SaveAndCloseBtn` and `TopicSelector`, both only defined in files summarizerPopup.html never loads. That's the underlying problem with the classic-script pattern - "does this global actually get loaded on this page" is tracked by hand (or not at all) instead of enforced by the module system.

Unlike the service worker split (mandated by Manifest V3) or the chrome.scripting.executeScript injected-page-context split (a real browser security boundary - see extension/summarizer.js), the classic-script split is not required by the platform. MV3 service workers already support `"type": "module"`, and popup/sidepanel HTML pages can equally load `<script type="module">` - background.js already proves ES modules work fine in this extension. This category of environment split is legacy from before the ES6 migration, not a platform necessity.

*** Scope
Convert popup.js, sidePanel.js, btContentScript.js, summarizerPopup.js, topicCard.js, topicSelector.js to ES modules with explicit import/export, following the same pattern Phase 1 established in app/. Once done, extension/'s ESLint config collapses from four classic-script blocks (one per HTML page) into a single ES-module block, matching app/'s. Cross-file references become explicit imports instead of implicit shared-scope globals, which would have made the SaveAndCloseBtn/TopicSelector bugs impossible to introduce rather than something a linter has to catch after the fact.

** The SessionManager project
The Session Manager adds a live "Current Session" hierarchy alongside the persisted topic tree. It introduces a dedicated `BTSessionNode` subclass and a new `sessionManager` module to mirror the browser's open windows, tab groups, and tabs inside the Topic Manager.

** Session nodes (`BTSessionNode`)
- Derive from `BTAppNode`, tagging themselves with `isSessionNode` and a `sessionType` of `ROOT`, `WINDOW`, `GROUP`, or `TAB` so existing helpers can continue to ignore them when looking up saved nodes.
- Carry browser identifiers (`windowId`, `tabGroupId`, `tabId`) rather than org IDs and expose only the subset of row actions that make sense (close, drag, todo). All other behaviors fall back to the base class only when appropriate.
- Use `BTSessionNode.generateWindowTitle()` to assign incremental window labels with active tab titles appended (`Window 1 - Google Search`, `Window 2 - GitHub`). Window titles update dynamically when users switch tabs.
- Implement proper color styling logic in `setTGColor()` override - session nodes only display tab group colors when represented in topic tree or are GROUP nodes themselves.
- Support bidirectional text synchronization with corresponding app nodes, ensuring notes are shared between persisted topics and live session state.

** Session lifecycle (`sessionManager`)
- Registers its handlers at startup and builds the session tree immediately after launch by processing the `launchApp` payload.
- Runs `syncToBrowser` after every tab/window/group event from the background script, reusing existing nodes, updating metadata (titles, colors, indices, favicons), and pruning anything no longer present in Chrome.
- Implements comprehensive text synchronization system that syncs notes between corresponding app and session nodes during browser snapshot reconciliation.
- Relays user actions performed on session nodes back to the browser through the existing `sendMessage`/`callBackground` pipeline (close tabs, move groups, toggle collapsed state), while app/topic nodes keep handling persisted org-mode data exactly as before.
- Uses time-based error retry prevention (2-second threshold) to avoid infinite sync loops when browser operations fail.
- Moved `requestBrowserSnapshot()` to `extensionMessaging.js` for better architectural separation and reuse across modules.

** Reconciliation Algorithm (`syncToBrowser`)
The core `syncToBrowser` function (~400 lines) performs a sophisticated multi-phase reconciliation:
1. **Preparation**: Groups tabs by window, creates lookup maps, initializes tracking sets
2. **Window/Group/Tab Processing**: Traverses browser snapshot, reuses or creates nodes, updates metadata (titles, colors, tab indices, favicons, active tab info for window titles)
3. **Text Synchronization**: Syncs notes from app nodes to corresponding session nodes using URL validation for TAB nodes and tabGroupId matching for GROUP nodes
4. **Pruning**: Removes session nodes for windows/groups/tabs that no longer exist in browser
5. **Ordering**: Sorts children by tabIndex to match browser tab strip order, updates group indices based on contained tabs
This approach minimizes unnecessary UI updates while ensuring the session tree always reflects current browser state and maintains text synchronization between persisted and live data.

** Text Synchronization System ✅ COMPLETED
Bidirectional text synchronization keeps notes in sync between persisted app nodes and live session nodes:

*** Synchronization Logic
- **TAB Nodes**: Validates URL match using `stripURLForComparison()` and `BTNode.compareURLs()` before syncing text
- **GROUP/TOPIC Nodes**: Matches by `tabGroupId` without URL validation since tab groups don't have URLs
- **Sister Nodes**: Enhanced `sisterNodes()` method finds corresponding nodes by `tabId` (TAB nodes) or `tabGroupId` (GROUP nodes)
- **Direct DOM Updates**: Updates `span.btText` innerHTML directly instead of calling `redisplay()` to preserve treetable structure

*** Synchronization Points
1. **Browser Snapshot Reconciliation**: `syncToBrowser()` syncs from app nodes to session nodes during browser state updates
2. **Edit Dialog**: `updateRow()` syncs text changes to sister nodes after user edits
3. **Tab Saving**: `bt.js` syncs text to sister nodes when existing nodes get note updates
4. **Error Handling**: Time-based retry prevention prevents infinite loops on sync failures

*** Technical Implementation
- Uses `stripURLForComparison()` to remove query parameters and fragments for reliable URL matching
- Direct DOM manipulation: `$(displayNode).find('span.btText').html(node.displayText())` 
- Validation prevents sync to mismatched URLs (e.g., when tabs navigate to different sites)
- Preserves treetable node references by avoiding `redisplay()` calls during sync

** Drag into the session tree ✅ COMPLETED
Verified working by testing.
- Drag of unopened appNodes into session opens tab or tab group in the dropped window. Only unopened nodes can be dropped. Dropped leaf/tab items are dropped onto the appropriate tabgroup if its open, otherwise its newly created.
- Drop into root session opens new window. Drop of topic opens new tab group w all tab items open.
- Implemented in tableManager: `dropNode()` intercepts the unopened-appNode-into-session case and routes to `openAppNodeInBrowser()`, which dispatches to `openAppNodeInWindow()` or `openAppNodeInTabGroup()`. `createSessionNode()`/`createBrowserWindow()` handle new window creation and group preservation. As anticipated, this uses existing BTAppNode methods and passes a windowId to force opening in a specific window.

** Drag of unsaved session nodes into the app tree - IMPLEMENTED, pending testing
This is not really a D&D, it's the creation of a new AppNode corresponding to a sessionNode - a copy, not a move. Session nodes don't serialize to org (orgText() returns '') and are owned by the syncToBrowser reconciler, so they can never be moved into the app tree. Instead the drop is intercepted in `dropNode()` (mirroring the app->session interception) and routed to `saveSessionNodeToApp()` in tableManager, which builds a payload from the session subtree and hands it to `saveTabs()` in bt.js - the same save machinery the Bookmarker uses. bt.js registers saveTabs via `registerSessionSaveHandler()` at startup to avoid an import cycle.

*** Behavior by dragged node type
- TAB: creates a new AppNode under the drop topic; the session node doesn't move. Standard save semantics apply in the browser via groupAndPosition: the tab is wrapped in a new tabgroup named for the topic if the topic has no open tabs, otherwise moved into the topic's existing tabgroup wherever it is. The payload passes groupId 0 - passing the tab's real group would make saveTabs adopt and rename the group the tab is sitting in (the Bookmarker popup avoids the same hazard by ungrouping before save; here the browser's group-move handles removal automatically).
- GROUP: creates a topic AppNode named for the tabgroup under the drop topic, with leaf AppNodes for each contained tab. The browser tabs are not impacted - a `grouping: false` flag added to saveTabs suppresses groupAndPosition. The real tabGroupId is passed so the new topic adopts the group (sister linkage). Suppressing grouping also suppresses the browser round trip whose tabGroupUpdated event normally delivers the tabgroup color, so the color is passed explicitly as `tabGroupColor` (taken from the session node) and applied to the topic after redisplay; setTGColor then propagates it to the contained tab rows.
- Already saved TAB or GROUP: re-files rather than saving a duplicate. `saveSessionNodeToApp()` looks up the existing appNode via `BTSessionNode.topicTreeNode()` and hands it to `moveNode()`, which moves it under the new topic and regroups its tab via handleNodeMove. Needed because saveTabs silently no-ops on a tabId it has already saved, so without this the drag would appear to work and do nothing.
- WINDOW/ROOT: drop does not take effect. Rejected in `BTAppNode.canAcceptDrop()` (via `BTSessionNode.canBeDroppedInTopicTree()`, a type check only), which also rejects any session drop onto trash (previously a session drag onto TRASH reached deleteNode).

*** Supporting details
- Session node notes are preserved: saveTabs now honors a per-tab `note` field overriding the common note, so annotated session tabs don't get blanked by the app->session text sync after save.
- Drops into the BOOKMARKS BAR topic trigger exportBookmarksBar() to resync the bar.
- After save, requestBrowserSnapshot() refreshes session rows to pick up inTopicTree styling.
- `moveSessionToApp()` and `moveAppToSession()` were deleted - both unreachable once dropNode intercepts the conversions. `moveNode()` now dispatches only the true moves: session<->session and app<->app. `BTSessionNode.canMoveTo()` is unchanged and correctly returns false for app-tree parents.
